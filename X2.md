# X2

X2 is a method for building production software with AI coding agents.

The [README](README.md) makes the case in plain language: code became the cheapest thing a
company owns, the decisions became the treasure, and the rules that must never break belong in
machinery rather than on paper. This document is the method itself: every rule it runs on, the
reasoning behind each one, and the mistakes that paid for them. It stands alone; you do not need
the README, or X2's predecessor
[X1](https://github.com/teklabsdigital/x1-method-for-claude-code), to follow it.

Much of what follows was learned the expensive way. X2 has run end to end twice: a pilot taken
from idea to one accepted slice under a formal acceptance test, and a second full product taken
from idea through public launch. In both, every human intervention was logged the moment it
happened and classified. That is a sample of two, and the metric section states its limits
plainly. Where a rule below exists because something went wrong, the document says what went
wrong.

## The three tiers

Most attempts to guide an AI assistant put the rules in prose: a standards document, a checklist,
a long prompt pasted at the start of every session. The assistant reads them again on every task.
They cost tokens every time they are read, they drift out of date, and nothing happens when one
is ignored. A rule the build does not check is only a suggestion.

So X2 sorts everything a method carries into three tiers and gives each a home where it cannot
decay.

**Invariants live in the build.** These are the rules that must always hold. There are 46 of
them, kept as claims in [`kernel/claims/`](kernel/claims/): one file per claim, naming the harm
it prevents, the mechanism that enforces it, and that mechanism's honest limits. The shipped
edition realises them as automated checks that run on every change; break one and the work simply
fails, whatever the AI says about it. A claim whose check is not yet built is tagged owed, with a
named trigger for when it must be, never silently assumed. Because the build carries them, the
invariants cost no prompt tokens and cannot be broken quietly.

**Decisions live in short files that people own.** The decomposition (a file called D-000, the
one decision about how the whole system divides up), the numbered decision records that follow it
(D-0xx), the deltas file, and the locked design prototype. This is the treasure, and it is kept
deliberately small.

**Consequences are everything generated from the decisions:** the code and its unit tests.
Disposable outputs, regenerated on demand. When X2 asks what a project produced, it points at the
gates passed and the exit reports, not at the code.

The unit of cost is human attention, counted in turns: one turn is one moment a person had to
step in, to read, decide or answer. The sole metric is human turns per shipped slice, where a
slice is a thin piece of the product that works end to end.

## X2 and the Agile Manifesto

X2 does not reject the Agile Manifesto, the 2001 statement of values that reshaped how software
teams work. It re-answers the Manifesto's 4 values for a setting the authors never had: an AI
agent doing the building. It keeps and sharpens 3 of them. The first, individuals and
interactions, it deliberately re-weights, and that re-weighting is the whole idea.

**Individuals and interactions over processes and tools.** This is the value X2 turns. An agent
has no lasting memory and no stake, so interactions cannot carry the rules that must always hold.
X2 puts those rules into a tool: the build. This is not "tools over people". The mechanical goes
into the tool so that people's attention is free for the decisions only people can make. The
result is less human process than Agile, not more, and no ceremonies to keep it running.

**Working software over comprehensive documentation.** X2 agrees about documentation and inverts
the asset. Working software is the proof; the decisions are the asset; the code is designed to be
regenerated from them on demand. X2 deletes the documents that describe the software, because
stored descriptions rot the moment the code moves. The only prose it keeps is the decisions a
machine cannot witness.

**Customer collaboration over contract negotiation.** X2 keeps the collaboration and changes the
artefact. The contract is a design prototype the customer approves, and the acceptance tests
derive from it. Judgement is captured as something executable, not negotiated as a document.

**Responding to change over following a plan.** X2 keeps this in full and aims to lower its
price. Because the code is treated as a disposable output of the decisions, changing a decision
and regenerating is designed to cost little; the first module regeneration is the named test of
that claim (DEC-1), and it has not yet run. Only three things must stay ahead of the work;
everything else stays cheap to reverse.

The through-line: Agile moved trust from documents to people. X2 moves the part of that trust
that must never vary from people to the build, and leaves the rest with people, where it belongs.

## Three things are decided up front

X2 is slice-first. Building starts before the requirements are finished, and most decisions wait
for the slice that needs them. Exactly three things are settled ahead of the work, because a
slice cannot cheaply reverse them:

1. **The decomposition, D-000.** One working session that looks at the whole solution space
   through five lenses and produces the one decision about how the system divides up.
2. **The persistent schema.** The shape of the data the product must keep. Data outlives code, so
   every change to the schema gets a human turn, without exception.
3. **The design spine.** The whole product's design system and information architecture,
   expressed as one prototype and designed once up front, because a coherent look and feel cannot
   be retrofitted across slices that are already built. The lock is per slice: each slice locks
   its own portion of the prototype as its acceptance contract and derives its tests from that
   portion alone. If the spine is revised, the affected portions are re-locked, and the approving
   turn is spent on the record.

Everything else stays reversible and waits for its slice. Regeneration is the reversal mechanism:
change the decision, regenerate what depends on it.

## The three gates

The pilot's sharpest process lesson was about approvals. They were extracted late, dragged out of
the person by interrogation, or simply assumed. So X2 fixes the exact moments a person is asked.

There are three gates. Each is a required, prompted turn that blocks the next phase:

1. **Stories approved.** Epic-level stories, one line each, define the problem domain. No
   acceptance criteria exist yet. Stories are dynamic: owners iterate them heavily, and there is
   an entire loop from the stories to the design prototype and back that refines the set until it
   is accepted. Gate 1 is provisional until that loop settles; re-approving a revised set is a
   normal gate turn, not a breach.
2. **Prototype approved, twice.** Claude Design, a separate design context (see the rig, below),
   produces the whole-product prototype, and the human approves it there: gate 2a. The prototype
   is then copied into the repository with its design system, verified, stamped with its
   provenance, and approved again: the lock, gate 2b. The fidelity tests hang off this turn. The
   lock also includes an exposure review, because a prototype can carry security-shaped choices
   nobody actually decided: an admin door drawn on a public page, a sequential identifier shown
   on a public surface. Faithful transcription would ship them. They are surfaced for a ruling at
   the lock, never silently transcribed.
3. **Ratified green.** Software teams call a build green when every automated check passes. The
   human reads the audited slice-exit report and accepts the slice. Green is decided here, on
   evidence, never in chat.

A phase that starts without its gate on record is a method breach, and the exit report must flag
it.

## The design phase

The design phase is three artefacts with separate owners and a strict order:

- the design brief
- the prototype, which is Claude Design's and never the builder's
- the derived tests, from the locked prototype only, never written ahead of it.

The pilot collapsed these into one step once and pointed them backwards, with the builder writing
acceptance criteria for its own work. The separation is now structural.

The design brief is requirements-for-design. For a simple domain it stays a thin note. For a rich
domain it is two things: the domain model frozen as a decision first, then a designer-facing
brief that renders that decision as user journeys, a screen inventory and the action edges that
link screens.

The brief's altitude is exact:

- **In scope:** topology (where the user goes and what they can do), shape (what each surface
  must expose) and priority (what the owner uses most).
- **Out of scope:** look (layout, components and palette belong to Claude Design) and behaviour
  under test (acceptance criteria derive from the lock).

The human approves the brief before gate 2a is requested. Claude Design's proposed information
architecture is then checked against it: every journey walkable, every named surface present, no
surfaces invented.

One scoping note. "No hand-authored test cases" bans fidelity tests written ahead of the lock. It
does not ban claim-driven tests (harness scenarios, gate tests). Those keep their write-first,
watch-it-fail shape, and they did the most catching in the pilot.

## The specification gap

A locked prototype pins the visual contract: what exists, where it lives, what it looks like. It
cannot express what happens when two people edit the same record at once, what an error path
should do, who is allowed to do what, or a domain's edge cases. X2 deleted hand-written
acceptance criteria, so it owes an answer for where that behaviour is specified.

The current answer is partial:

- claim-driven tests cover what the kernel claims: security, tenancy, data, time, wire
- the domain model frozen for the design brief covers structure, not behaviour
- everything else surfaces as a record-gap question (a D-0xx) during implementation, or as
  green-but-wrong at review.

The last two depend on a person noticing. That makes this the method's largest open gap: a thin
behaviour tier is owed between the one-line stories and the locked prototype, for the rules a
prototype cannot show. It is listed in "What remains open".

## The twelve skills

A skill is a short instruction file the agent loads when a job begins. Twelve of them carry the
method, and they are short by design: the kernel already enforces what a prompt would restate, so
a skill earns length only by what the build cannot check. No skill specifies commodity the model
already knows, such as how to write pagination.

In workflow order:

- **x2:stories**: epic stories, gate 1
- **x2:seed**: create the repo and anchor the standing constraints, before any decision work
- **x2:decompose**: D-000, the schema decisions, the deltas file, the launch pre-flight
- **x2:design**: the whole-product prototype, in Claude Design's own context
- **x2:lock**: copy the prototype down and lock the slice's portion, gate 2b
- **x2:derive-tests**: the fidelity ledger and acceptance scenarios, from the locked portion
- **x2:implement**: build the locked portion to green, autonomously
- **x2:slice-exit**: the report that defines green
- **x2:problem-solve**: runtime diagnosis, priced in human turns
- **x2:extract**: the post-project extraction loop, folding the ledger's evidence back into the
  method
- **x2:help**: the index
- **x2:adopt**: migrate an existing project onto X2.

Two of these earn their place from the pilot's costliest gaps. x2:seed exists because missing
scaffolding cost about a quarter of the pilot's turns. x2:lock exists because the pilot treated
the prototype as locked without ever asking anyone.

## Green is a report

Green is declared by the slice-exit report, audited before ratification. Never by prose, never in
chat.

The report states:

- every test tier run this session, with counts, and stale results marked stale
- the composed-app smoke: the real entry point of the real application makes at least one real
  request per primary flow
- the service-method coverage table, with gaps named
- unsurfaced decisions, which must be none
- pending rulings
- the lock status
- the slice turn count, from the ledger.

The smoke test exists because the pilot went "all green" with a sign-in button that did nothing.
Every piece passed its own tests; the assembled product did not run. Pieces passing their own
tests is not the product running, so the report must show the product running.

What the builder owes between gates 2 and 3 is all of the locked portion, with no silently
dropped pieces. "All of it" binds to the locked portion, not the whole prototype; the rest of the
spine awaits its slices.

## The metric, and its honest limits

The metric is human turns per shipped slice:

- **Counted:** turns the method forced, split into sanctioned (decisions, green-but-wrong
  adjudications) and defect-shaped (edition defects, invariant gaps, method defects).
- **Not counted:** admin turns, ideation turns, and the up-front investment (D-000, the schema,
  the design spine).

One counting rule governs the whole history; the reconciled figures live in
[`record/metrics.md`](record/metrics.md). The metric's limits, stated plainly:

- There is no baseline yet. Nobody has built the same product without the method and counted.
- Slices are not standard units, so the figure can flatter itself through bigger slices.
  Per-slice mid-build cost and defect share travel across projects; whole-project totals do not.
- It does not capture wall-clock time, token cost or outcome quality.
- Builder and intake are separate contexts of the same model. The separation removes
  self-grading; it does not remove shared blind spots.

**The figures so far** (two projects; the record layer holds the reconciliation):

- The pilot's only slice cost about 18 counted turns under the current rule (25 under the
  acceptance test's original wider count), with a defect share of about a third, a quarter of all
  turns spent on bootstrap, and "all green" declared twice before it was true.
- The second project shipped five slices by exit report (a sixth built green, report owed). The
  first slice held roughly flat like for like; every subsequent slice cost 1 to 7 mid-build
  turns, one of them zero. Defect share fell to about a tenth, and the pilot's watch classes went
  to near zero: bootstrap 1, green-but-not-running 0, completeness interrogations 0.

The reading: the first slice is where the method charges its fixed cost; the marginal slice is
where it pays out. The second project's turn mass moved out of the slices into phases the method
had no names for; naming them is what the extraction loop added (see below).

## The launch phase

Applies when the product ships to a public audience. The second project's largest cluster of
turns that were nobody's defect was launch work arriving as ad-hoc directives at the end: brand
chrome, legal pages, discoverability, social presence, analytics, abuse pricing, and edge
hardening. None of it was a mistake; all of it was unbudgeted.

So the method names the phase and moves its decisions forward:

- **The launch pre-flight** runs at decompose: rule each launch concern in or out early,
  including the estate question, because a greenfield app can land inside an existing brand
  estate (styles, domains, policies) that is not greenfield.
- **Exposure values are ruled before first use.** Serving domains, sender addresses, public
  identifiers, entry points: assumed values propagate into green artifacts and surface late.
- **The lock includes an exposure review** of the frozen artifact, since a prototype can carry
  security-shaped choices that transcription would ship.

## The extraction loop and negative space

After a project completes, x2:extract folds its evidence back into the method. The turn ledger is
read as a map of negative space: every human turn marks a place the method was not yet good
enough, whether or not anything visibly broke.

Defect turns are the loud half. The quiet half is churn. A wrong-but-green decision leaves no
defect behind it; it gets reworked across many small turns, each one individually fine, all of
them clustering on the same decision. No retrospective surfaces this, because nothing ever went
red. The extraction computes turn-count per decision, and high-churn, zero-defect decisions are
nominated as negative-space candidates.

Churn nominates; it never judges. A human rules every nomination into one of two classes:
**iteration-inherent** (the churn is the work: story refinement, model selection, content voice;
the method's job is to name the loop so it is budgeted) or **pre-decidable** (the churn was
avoidable; the method pre-flights it next time). The ruled register and the cross-project churn
table live in [`record/negative-space.md`](record/negative-space.md), and decompose reads that
table as a pre-flight: the method anticipating absence, not only recording it.

Two disciplines guard the loop. Everything that crosses into the method is sanitized: the method
receives the lesson, never the case file; raw evidence stays in each project's private
repository. And every crossing item carries an applies-when condition, because projects come from
many verticals and nothing is universal unless it says so. One limit is admitted: churn cannot
catch the dormant-wrong, a bad decision made once and never revisited. For the chronic churn
types the register marks, decisions record a confidence and a wrong-if line, so reality can trip
them later.

## One law: anything that must survive lives in a file

An AI assistant's working memory is finite. When a long session fills it, the history is
compressed to make room, and detail falls out. X2's one law is that memory like that is not
storage: a compactible context is never trusted to hold anything durable. Two disciplines follow
from it, and the pilot measured the price of lacking both.

**The ledger.** The pilot's builder held the turn count only in its working memory. Compression
destroyed the first third of the record. So now every human turn is appended to a ledger file in
the repository the moment it happens: the number, what happened, why a person was needed, and its
bucket. Every turn is also fed to intake as it happens; intake classifies, and its bucketing
governs. Even the append rule itself is not trusted to memory: the edition ships a hook that
injects the ledger reminder on every human prompt, because the second project showed that a rule
anchored in the repository alone still drifts in long sessions.

**The constraints.** The pilot carried its standing rules in a pasted handover note. Compression
dropped them, 55 breaches of one non-negotiable rule accumulated silently, and the first
self-check declared them clear when they were not. So now, at seed time, the constraints are
written into the repository's CLAUDE.md, which re-enters every session. Where a constraint is
mechanically checkable, the check ships beside it.

Anything that must survive the session lives in a file.

## The rig: separate contexts

X2 runs as separate contexts, each a distinct AI conversation with its own memory and its own
job:

- the **builder** drives the slice and is the measured subject
- **intake** owns the ledger and the record, classifies turns and audits the exit report
- **Claude Design** owns the prototype
- the **author** (in the method's own development) owns the skills and this document.

This is not ceremony. Each separation earns its place:

- Builder and designer are separate because a builder who designs collapses the design steps and
  writes acceptance criteria backwards, grading its own homework twice over. The pilot did
  exactly this at turn 8.
- Builder and intake are separate because the measured subject must not grade its own buckets.
  Intake's independent ledger recovered what the builder's compressed memory lost, three times.
- Builder and ratification are separate for the same reason green is a report: the builder
  declares with evidence, intake audits, and the human reads one document.

## Prior art

The enforcement half of X2 builds on known ideas: architecture fitness functions, ArchUnit-style
tests (the shipped edition uses NetArchTest), module-boundary rules and policy-as-code. The
workflow half sits beside the spec-driven wave: GitHub Spec Kit, Kiro and similar.

What X2 adds is the sorting and the measurement discipline:

- the three-tier sort, with each tier placed where it cannot decay
- human turns per shipped slice as the first-class metric, kept in a ledger
- green as an audited report, never prose
- a conformance vocabulary that admits what is proven, patterned, latent or owed.

If you know those tools, read X2 as packaging and measurement on top of them, not a new invention
of build-time enforcement.

## What remains open

The invariants pass (2026-07-11) closed most of what the acceptance test left open:

- the cooling-off window is 90 days, with the number held in one place
- the locked artefact is the design source alone, with the design system under
  `design/prototype/_ds/`
- the thin-UI import ban and the harness completeness rule are enforced by the kernel (UI-5,
  TEST-2), not held by discipline.

The extraction pass (2026-07-21) closed two more: the record question (the method publishes the
sanitized distillate under [`record/`](record/); raw project records stay private by ruling) and
the missing feedback loop itself (x2:extract, with the negative-space register and the
cross-project churn table).

Still owed, each with a named trigger:

- the behaviour tier for the specification gap (trigger: the first slice whose core value a
  prototype cannot show, such as a permission matrix or a concurrency rule)
- the first regeneration of a module from its decisions, with what broke reported. This single
  experiment decides whether the regeneration thesis is real (DEC-1)
- a baseline run: the same product built without the method, turns counted, both results
  published. Until it exists, the figures are a starting line, not a comparison
- a second edition for another stack, which is what turns "technology-neutral" from intent into
  fact
- the prototype-to-ledger exporter that makes the fidelity ledger single-source (UI-4)
- the shared design-system home (trigger: a second product reusing the same system)
- the catalogue's standing owed list.

Known frictions, named rather than hidden:

- gate 2 currently binds to one design tool and its export format, and the cost of re-locking
  later slices against a revised spine is unmeasured
- the rig is hand-operated: builder, intake and designer are separate contexts a solo adopter
  runs by hand.

Nothing here blocks running the method.

## Verification

The skills are right if a new project reaches a green first slice with only:

- one D-000 file
- one deltas file
- a locked prototype
- derived tests
- a ledger
- an exit report

and no architecture document, no system-wide requirements phase and no hand-authored fidelity
tests.

This document is right if a reader can build the method's model, and its rationale, from it
alone. The figures to beat live in [`record/metrics.md`](record/metrics.md).
